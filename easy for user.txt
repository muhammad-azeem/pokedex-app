PART 1 — WHERE DO THE POKEMON COME FROM?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

There is a FREE public website (called an API) that gives
Pokémon data to anyone who asks:

    URL: https://graphql-pokemon2.vercel.app/

You ask it a question like:
  "Give me the first 151 Pokémon with their name, number, image and types"

It replies with data in JSON format (like a structured list).

This is called a GRAPHQL API.
GraphQL = a way to ask for exactly the data you need, nothing more.

We do NOT store any Pokémon data ourselves.
Every time the page loads, our server asks this API and shows the result.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PART 2 — WHERE DO THE IMAGES COME FROM?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The API also gives us image URLs for each Pokémon.
Example: https://img.pokemondb.net/artwork/bulbasaur.jpg

That image URL points to a website called "pokemondb.net"
which hosts official Pokémon artwork for free.

So the flow is:
  1. Our app asks the GraphQL API for Pokémon data
  2. The API returns a list including image URLs
  3. Our app puts those URLs into <img> tags
  4. The browser downloads images directly from pokemondb.net

We do NOT download or store any images ourselves.

We use Next.js's built-in Image component (next/image) which:
  - Automatically resizes images for different screen sizes
  - Lazy-loads images (only loads what's visible on screen)
  - Converts images to modern formats (WebP) for speed

We had to tell Next.js that "img.pokemondb.net" is a trusted
image source in next.config.ts — otherwise it would block it.

PART 3 — HOW DOES THE APP TALK TO THE API?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: lib/api.ts

This file contains two functions:
  fetchPokemons(151)   → gets the list of 151 Pokémon
  fetchPokemon("bulbasaur") → gets full details of one Pokémon

How it works technically:
  We send a POST request (like filling a form and submitting it)
  to the GraphQL API URL with our question written in GraphQL
  language inside the request body.

Example question we send (called a "query"):
  query Pokemons($first: Int!) {
    pokemons(first: $first) {
      id, number, name, image, types
    }
  }

The API reads this and replies with exactly those fields.

We wrote our own fetch function (not a library) because the
graphql-request library version 7 had a bug causing 405 errors.
Using plain fetch gave us full control.

PART 4 — HOW DOES THE PAGE GET BUILT? (Next.js App Router)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This app uses Next.js with the "App Router" (newest style).

Think of it like a restaurant kitchen:
  - Server Components = the kitchen (does heavy work, user can't see it)
  - Client Components = the waiter (interacts with the user at the table)

HOMEPAGE ( app/page.tsx )
  - This is a SERVER component
  - When you open the website, the server:
      1. Calls fetchPokemons(151) to get all 151 Pokémon
      2. Builds the HTML page with the data already in it
      3. Sends the finished HTML to your browser
  - Your browser receives a complete page immediately
  - No waiting for JavaScript to load and fetch data

DETAIL PAGE ( app/pokemon/[name]/page.tsx )
  - Also a SERVER component
  - The [name] in the folder name means it works for ANY Pokémon
  - Visit /pokemon/bulbasaur → fetches Bulbasaur's details
  - Visit /pokemon/pikachu  → fetches Pikachu's details
  - If the name doesn't exist → shows a 404 "Not Found" page

INTERACTIVE PARTS ( components/PokedexClient.tsx )
  - This is a CLIENT component (runs in the browser)
  - It handles: search box, type filter buttons, pagination
  - It receives the 151 Pokémon list from the server as a "prop"
    (like passing data in an envelope)
  - All filtering happens instantly in the browser — no new
    network requests needed

PART 6 — HOW SEARCH AND FILTER WORK
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The server already sent all 151 Pokémon to the browser.
So searching and filtering is INSTANT — no internet needed.

When you type "bulb" in the search box:
  - JavaScript on your page filters the 151 Pokémon array
  - Only shows ones where name includes "bulb"
  - No new request is made to the API

When you click a type like "Fire":
  - Same thing — filters the local array
  - Shows only Fire-type Pokémon

Pagination (20 per page):
  - We slice the filtered array: page 1 = items 0-19, page 2 = items 20-39
  - Changing pages is instant too

PART 7 — HOW FAVORITES WORK
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

When you click the heart ♡ on a detail page:
  - The Pokémon's ID is saved to your browser's "localStorage"
  - localStorage is like a mini notepad inside your browser
  - It stays saved even after you close the tab
  - It is NOT saved on any server — only on your device

Next time you visit that page → we check localStorage → heart shows ♥

PART 8 — HOW COLORS WORK (Type Colors)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: lib/typeColors.ts

We have a manual dictionary (lookup table) that maps each
Pokémon type to a color:
  Fire    → orange
  Water   → blue
  Grass   → green
  Psychic → pink
  Ghost   → dark purple
  ... etc for all 18 types

This is used in:
  - Card headers (gradient background matches the type)
  - Type badge pills (colored labels on cards)
  - Detail page hero (background gradient)

PART 9 — LOADING AND ERROR STATES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

LOADING STATE:
  While the server fetches from the API, Next.js shows the
  loading.tsx file automatically. It shows grey placeholder
  boxes (called "skeletons") in the shape of the real content.

ERROR STATE:
  If the API call fails (network down, API broken), Next.js
  shows error.tsx automatically. It shows a message and a
  "Try Again" button.

NOT FOUND (404):
  On the detail page, if you visit /pokemon/fakename123
  the code calls notFound() which triggers not-found.tsx.
  This also sends a real HTTP 404 status (important for SEO).

PART 10 — INTERVIEW QUESTIONS YOU CAN ASK THE ENGINEER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ABOUT THE API:
  Q: "Where does the Pokémon data come from?"
     A: A public GraphQL API at graphql-pokemon2.vercel.app

  Q: "Why GraphQL instead of REST?"
     A: GraphQL lets you ask for exactly the fields you need.
        For the list page we only need name/image/types.
        For the detail page we need attacks/evolutions/stats.
        With REST you'd get all fields every time (wasteful).

  Q: "Why did you use plain fetch instead of graphql-request?"
     A: graphql-request v7 had breaking changes causing 405 errors.
        Plain fetch gives full control over headers and method.

ABOUT NEXT.JS:
  Q: "Why did you keep page.tsx as a Server Component?"
     A: The assessment explicitly required it. Server Components
        fetch data on the server — the user gets a full HTML page
        immediately without waiting for JavaScript to run.

  Q: "Why is PokedexClient a Client Component?"
     A: Because it uses useState for search, filter, and page.
        useState is a React hook that only works in the browser.

  Q: "What happens when a user visits /pokemon/pikachu?"
     A: Next.js matches the [name] dynamic route, the server
        calls fetchPokemon("pikachu"), gets the data from the
        API, builds the HTML, and sends it to the browser.

  Q: "What if someone visits /pokemon/fakemon123?"
     A: fetchPokemon returns null, the code calls notFound(),
        Next.js shows the 404 page with a real HTTP 404 status.

  Q: "How does the search work — does it call the API on every keystroke?"
     A: No. All 151 Pokémon are already in memory from the
        initial server fetch. Search just filters that array
        locally in the browser. Zero network calls.

ABOUT IMAGES:
  Q: "Where do the Pokémon images come from?"
     A: The API returns image URLs pointing to img.pokemondb.net.
        We didn't host any images ourselves.

  Q: "Why use next/image instead of a regular <img> tag?"
     A: next/image automatically optimizes images — it resizes
        them for the user's screen, converts to WebP format,
        and lazy-loads them. Better performance automatically.

  Q: "Why did you add img.pokemondb.net to next.config.ts?"
     A: Next.js blocks external images by default for security.
        You must whitelist trusted domains explicitly.

ABOUT TYPESCRIPT:
  Q: "Why TypeScript?"
     A: It catches mistakes before runtime. For example, if the
        API changes and removes a field, TypeScript will show a
        red error in the editor immediately.

  Q: "Where are the types defined?"
     A: lib/types.ts — all API response shapes are typed there.
        PokemonListItem for the grid, PokemonDetail for the
        detail page, PokemonAttack, PokemonEvolution, etc.

ABOUT DESIGN:
  Q: "How did you decide which color to use for each Pokémon type?"
     A: We have a manual lookup table in lib/typeColors.ts
        mapping each of the 18 types to Tailwind CSS classes.

  Q: "Is the app responsive?"
     A: Yes. The grid uses Tailwind breakpoints:
        2 columns on mobile → 3 on tablet → up to 6 on desktop.

  Q: "Does the app support dark mode?"
     A: Yes. Every component has dark: variants in Tailwind
        that activate when the OS is in dark mode.

TRICKY QUESTION TO ASK:
  Q: "The original API URL in the spec was graphql-pokemon.vercel.app
     but you used graphql-pokemon2.vercel.app — why?"
     A: Because the original URL now returns 405 Method Not
        Allowed on POST requests — it was replaced by a landing
        page SPA. I discovered this by testing the endpoint
        directly with curl/PowerShell and found the v2 URL
        still works. I documented this in the README trade-offs.

QUICK SUMMARY (30 seconds version)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. User opens the website
2. Next.js server asks a free public API for 151 Pokémon data
3. API replies with names, numbers, types, and image URLs
4. Server builds the HTML page and sends it to the browser
5. Browser shows the Pokémon grid with images from pokemondb.net
6. User searches/filters → happens instantly in the browser
7. User clicks a card → server fetches that Pokémon's full details
8. Detail page shows stats, attacks, weaknesses, evolutions
9. User clicks heart → saved to their browser's localStorage

Everything is typed with TypeScript.
Everything is styled with Tailwind CSS.
No database. No login. No backend of our own.
Just a Next.js app talking to a free public GraphQL API.
